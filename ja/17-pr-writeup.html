TL;DR

通知の送信はリクエストパスの中でインラインに実行されていました。負荷がかかると、これがミューテーションのレイテンシーに200〜800 msを追加し、SMTPプールが枯渇した際にはメールを黙って取り落としていました。このPRでは、配信を既存のpg-bossキューに移行することで、APIが即座にレスポンスを返し、失敗した送信はバックオフしながらリトライされるようにします。

背景

先の四半期にコメント内での@メンション機能を追加した際、メンションが発生するたびに、コメントを保存する同じトランザクション内で最大3件の送信(アプリ内通知、メール、Slack)がトリガーされるようになりました。リリース当初は問題ありませんでしたが、今では1件のタスク更新が40人のウォッチャーにファンアウトすることもあり、もはや問題なしとは言えません。

変更前
  • 送信はミューテーションハンドラー内でインラインに実行される
  • SMTPタイムアウト = コメント自体が500エラーになる
  • リトライなし — 取り落としたメールは戻らない
  • comments.createのp99: 1.4 s
変更後
  • ハンドラーは受信者ごとに1件のジョブをキューに登録し、即座にレスポンスを返す
  • ワーカーは指数バックオフで3回リトライする
  • リトライを使い果たした後の調査用にデッドレターテーブルを用意
  • comments.createのp99: 180 ms(ステージング)

packages/notify/src/worker.ts 新規 +126

このPRの中核。notify.deliverジョブを取り出し、ユーザーのチャネル設定を解決して、適切なアダプターを呼び出すpg-bossのサブスクライバーです。リトライはチャネルごとに設定されており、メールは3回試行、Slackは1回のみです。これはSlack側のAPIがすでに冪等になっているためです。

boss.work('notify.deliver', { batchSize: 20 }, async (jobs) => {
  for (const job of jobs) {
    const { userId, event, channel } = job.data;
    const prefs = await getPrefs(userId);
    if (!prefs[channel]) return;          // user muted this channel

    try {
      await adapters[channel].send(userId, event);
    } catch (err) {
      if (job.retryCount >= MAX_RETRY[channel]) {
        await deadLetter(job, err);         // don't throw — ack & park
        return;
      }
      throw err;                            // pg-boss reschedules
    }
  }
});
