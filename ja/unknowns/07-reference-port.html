1 クレートが実際に行うこと
  • トークンバケットによる許可制御(bucket.rs): 送信されるすべてのリクエストは、rate_per_sec = 8 で補充され burst_capacity = 40 を上限とするバケットから1トークンを引き出します。バケットは満杯の状態で開始するため、コールドスタートのクライアントは即座に40リクエストをバーストできます。
  • 補充は遅延評価かつ整数切り捨て: トークンは、経過ナノ秒 × レート ÷ 10⁹ から、リクエストが到着したときにのみ生成されます。小数トークンは一切保存されません — 代わりに、last_refill は1トークン分が丸ごと生成されたときにのみ進み、トークン未満の経過時間は次回へ持ち越されます。
  • 非相関ジッターバックオフ(backoff.rs): リトライ可能な失敗が発生すると、次の遅延は base = 250ms、cap = 30s として [base, min(prev × 3, cap)] の範囲から一様に選ばれます。抽選結果はそれぞれ次回の種になります — 遅延は一斉に倍増するのではなくランダムに揺らぐため、障害後にクライアント同士の同期がずれます。
  • リトライバジェット(budget.rs): これは2つ目の、別のバケットです。初回試行が成功するたびに1を預け入れ、リトライを試みるたびに10を引き出します。残高が引き出しに足りない場合、リトライは完全に拒否されます（初回試行のリクエストはそれでも送信されます）。これにより、継続的な障害発生時のリトライ増幅は約10%に抑えられます。
  • 失敗の分類は呼び出し側の責務: このクレートはエラーの内容を一切検査しません。acquire()、next_delay()、try_withdraw() を公開し、何をリトライ可能とみなすかは呼び出し側が決定します。TS版のポートもこの境界を保持します — src/api/errors.ts にあるAcmeの isRetryable() が唯一の判断基準であり続けます。

1時計が逆行する場合。 saturating_duration_since は負の経過時間をゼロにクランプします。Date.now() はNTP補正によって時刻が後退することがありますが、ポート版は単調増加する performance.now() を使用し、さらに Math.max(0, …) を二重の安全策として保持しています。
2整数演算と浮動小数点演算の違い。 Rustの u64 除算は切り捨てられますが、JSの除算は切り捨てられません。Math.floor で切り捨てを再現しています。ナノ秒ではなくミリ秒を使うのも安全です: rate = 8 のとき、elapsedMs * 8 は 2⁵³ よりずっと小さい値に留まるため、精度が失われることはありません。
3構造を支える重要なガード。 last_refill は、丸ごとのトークンが生成されたときにのみ進みます。このガードを外す（つい行いたくなる「簡略化」）と、呼び出しごとにトークン未満の進捗が黙って捨てられてしまいます — 低レートの場合、頻繁なポーリング下ではバケットが永遠に補充されなくなります。そのため完全に保持し、リグレッションテストも追加しています。

4飽和乗算。 saturating_mul(3) は u64 のオーバーフローを防ぎます。JSではオーバーフローが発生しません — capMs = 30_000 によって積は 2⁵³ に達するはるか手前で制限されるため — このガードは意図的に外しています（§3の「削除」列を参照）。
5両端を含む範囲か含まない範囲か。 Rustの lo..=hi は両端を含みます。素朴な lo + random() * (hi - lo) では hi が出ることは決してありません。Math.floor 内の + 1 が両端を含む挙動を再現しています — 1文字違いでバグを生みやすい箇所なので、承認するか却下するか判断できるようここで明示しています。
6状態を持つシード。 各回の抽選結果が次の反復の prev になります — これによって、単純なジッター付き指数バックオフではなく非相関な挙動が生まれます。reset() は成功時に prevDelayMs = baseMs へ戻し、クレートの Backoff::reset と一致します。this.random は決定的なテストのために注入可能です（クレート側もテストでシード付きの SmallRng を使用しています）。

tryWithdraw(): boolean {
  // no lock: single-threaded event loop7 —
  // but NO await between check and debit.
  this.depositDrip(this.clock());
  if (this.balance >= WITHDRAW_COST) {
    this.balance -= WITHDRAW_COST;8
    return true;
  }
  return false9;
}
7スレッド安全性 → イベントループ。 クレートの Mutex が存在するのは、Rust側の呼び出し元がワーカースレッドから引き出しを行うためです。Acmeのクライアントは単一のイベントループ上で動作するため、このロックはなくなります — ただし、それが提供していた原子性は規約として維持する必要があります: tryWithdraw は最初から最後まで同期的に動作させ、eslintの no-await-in-budget 境界コメントを追加するとともに、このメソッドがPromiseを返さないことを検証するテストも追加します。
8チェックと引き落としを一体化。 両側とも同じ形です。ここでの故障モードはデータ競合ではありません — 将来の改修でチェックと引き落としの間に await（例えばログ出力のため）が挿入され、進行中の2つのリトライが両方ともチェックを通過してしまうことです。
9拒否するだけで、キューには入れない。 バジェットが空の場合、クレートは即座に false を返します — リクエストは初回試行として送信されるか、即座に失敗するかのいずれかです。ポート版は、リトライを後回しにするために決して「親切心」でキューに入れてはいけません。それをすると、バジェットが本来防ぐべきリトライの暴走を再び引き起こしてしまいます。
3 挙動: 保持 / 意図的な変更 / 削除
完全に保持
  • 補充の切り捨てとガード — 丸ごとのトークンのみ；lastRefill はトークン生成時のみ進む
  • ジッターの計算式 — [base, min(prev×3, cap)] の範囲で一様分布、両端含む
  • バジェットの経済性 — 初回成功ごとに+1、リトライごとに−10、上限1000
  • バケットは満杯状態で開始 — 40リクエストのコールドバーストは意図的な仕様（クレートのテスト burst_at_t0 と一致）
  • 呼び出し側がエラーを分類 — リミッターは失敗内容を検査しない
意図的に変更
  • Instant → performance.now() — どちらも単調増加；レート8/sではミリ秒の精度で十分
  • u64 のナノ秒 → number のミリ秒 — すべての積が2⁵³未満であることが保証される；Math.floor が整数除算を再現
  • Mutex<Budget> → 単純なフィールド — 原子性は「同期処理のみ」という規約とテストで担保
  • SmallRng → 注入された random() — デフォルトは Math.random、テストではシード付きスタブを使用
削除（不要）
  • saturating_mul のオーバーフローガード — ミリ秒の範囲でcapが適用されれば到達不可能
  • Send + Sync の実装、Arc のクローン — 共有すべきスレッドが存在しない
  • tokio/async-std のフィーチャーフラグ — ポート版は構造上ランタイムに依存しない
  • Prometheusのカウンター — Acmeでは代わりに telemetry.track() で送信；フック地点は維持
