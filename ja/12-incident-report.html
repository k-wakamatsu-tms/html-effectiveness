INC-2025-0412

タスク同期における502エラーの増加

SEV-2 解決済み 継続時間 47分 検知日時 4月12日 · 14:07 担当者 Devon Park
概要

設定のロールアウトにより、sync-worker ティアのデータベース 接続プール上限が64から8に引き下げられ、通常の午後の負荷の下で接続が 枯渇しました。同期APIは47分間にわたり、リクエストの約21%で502エラーを 返しました。設定を元に戻し、ワーカーフリートを再起動することで軽減対応を 行い、データの損失はありませんでした。

タイムライン

14:02
設定変更 cfg-9a12 が、通常のロールアウト パイプラインを通じて本番環境に反映されました。
14:06
影響が発生。 同期ワーカーがプールのチェックアウトで キューイングを開始。p95レイテンシーが4秒を超えて上昇し、 ロードバランサーが502エラーを返し始めます。
14:07
アラート発報：sync_5xx_rate > 2% が60秒間継続。 オンコール担当（Devon）が対応を開始。
14:18
当初の仮説はAPIサービスの不良なデプロイであり、直近2件の アプリケーションデプロイをロールバックしたが効果はありませんでした。
14:31
Miraが参加し、ワーカーダッシュボードでプール待機の飽和状態に 気づきます。調査の焦点はアプリコードではなくインフラ設定へと 切り替わります。
14:44
軽減対応完了。 cfg-9a12 をロールバックし、ワーカーフリートを再起動。5xxエラー率は3分以内に 0.2%未満まで低下しました。
14:49
モニターが5分間正常な状態を維持。インシデントは解決済みと宣言され、 ステータスページが更新されました。

根本原因

PR #4888 により、接続プールの上限を ワーカーティアごとに設定可能にしました。新しい sync-worker キーのデフォルト値はグローバル値 （64）を継承する想定でしたが、ローカルテスト中に8にハードコードされたまま コミットされてしまいました。設定リンターは型のみを検証し、値の大きさは 検証しないため、この変更はCIを通過しました。

設定のロールアウトとコードデプロイは別々のパイプラインを通るため、 オンコール担当が最初にとった対応 — 直近のアプリケーションデプロイの ロールバック — は効果がなく、診断に約13分を要する結果となりました。

infra/config/workers.yaml
pool:
global_max_connections: 64
tiers:
- sync-worker: { max_connections: 64 }
+ sync-worker: { max_connections: 8 } # debug value, do not ship
webhook-worker: { max_connections: 32 }

影響範囲

失敗したリクエスト数（502） ~41,200
ピーク時エラー率 21.4%
影響を受けたユーザー ~2,300ワークスペース
データ損失 なし — クライアント側で再試行
SLA違反 なし（月間バジェット内）

アクションアイテム

DP cfg-9a12をロールバックし、プール上限を64に復元 4月12日
MO max_connections に対する設定リンターの範囲チェックを追加（32未満で警告） 4月18日
SR オンコールダッシュボードにデプロイと並べて“直近の設定ロールアウト”を表示 4月25日
PA フリート全体への適用前に、設定変更を1つのワーカーAZに10分間カナリア展開 5月2日
オンコールメモとアラート履歴をもとに作成 · レビュー担当：Devon Park、Mira Okafor
