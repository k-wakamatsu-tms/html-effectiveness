acme/web ・ アーキテクチャノート

コードベース内における認証の流れ

Acmeはクッキーベースのセッションを採用しており、ブラウザがベアラートークンを直接保持することはありません。認証が必要なリクエストはすべて/api/*にアクセスし、単一のverifyToken()ミドルウェアを通過して、下流のハンドラーがreq.ctxから読み取るSession行に解決されます。このミドルウェアだけがセッションストアとやり取りし、そのセッションストアだけがsessionsテーブルとやり取りします——つまり、検討すべき信頼境界はただ一つしかありません。

src/app/providers/AuthProvider.tsx :22-48

マウント時に、Reactプロバイダーはcredentials: 'include'を指定してGET /api/sessionを発行し、fw_sidクッキーが一緒に送信されるようにします。レスポンスはcurrentUserをコンテキストにハイドレートするか、nullのままにします。後者の場合、ルーターはそれを「サインイン画面を表示する」ものとして扱います。

ソースを表示 
// src/app/providers/AuthProvider.tsx
export function AuthProvider({ children }: Props) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetch('/api/session', { credentials: 'include' })
      .then(r => r.ok ? r.json() : null)
      .then(setUser);
  }, []);

  return <AuthCtx.Provider value={{ user }}>{children}</AuthCtx.Provider>;
}

3
src/middleware/auth.ts :14-31

ここが信頼境界です。verifyTokenは署名付きのfw_sidクッキーを読み取り、SessionStoreにその解決を依頼し、req.ctx.sessionを設定するか401を返します。アプリ内の保護されたルートはすべてこの関数の背後にマウントされているため、その挙動を変更すると認証全体に影響します。

ソースを表示 
// src/middleware/auth.ts
export async function verifyToken(req, res, next) {
  const raw = req.signedCookies['fw_sid'];
  if (!raw) return res.status(401).end();

  const session = await SessionStore.get(raw);
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).end();
  }

  req.ctx = { session };
  next();
}
4
src/lib/sessionStore.ts :8-52

SessionStoreは小さなリードスルーキャッシュです。まずプロセス内のLRUを確認し、なければPostgresにフォールバックします。書き込み(create、revoke)は常にDBへ直接行われ、キャッシュエントリを無効化するため、他のワーカーが古いセッションを返すことはありません。

ソースを表示 
// src/lib/sessionStore.ts
const cache = new LRU<string, Session>({ max: 5000, ttl: 60_000 });

export const SessionStore = {
  async get(id: string) {
    const hit = cache.get(id);
    if (hit) return hit;
    const row = await db.one(SELECT_SESSION, [id]);
    if (row) cache.set(id, row);
    return row ?? null;
  },
  /* create, revoke, touch ... */
};
5
db/migrations/004_sessions.sql :1-18

sessionsテーブルはランダムな32バイトのid(クッキーの値)をキーとしており、「どの端末からでもサインアウト」を実現するためにuser_idへのカバリングインデックスを持っています。有効期限はここ(expires_at)と、多層防御としてミドルウェアの両方で強制されます。

ソースを表示 
-- db/migrations/004_sessions.sql
create table sessions (
  id          text primary key,
  user_id     uuid not null references users(id),
  created_at  timestamptz default now(),
  expires_at  timestamptz not null,
  ip          inet,
  user_agent  text
);
create index sessions_user_id_idx on sessions(user_id);

主要ファイル

  • src/middleware/auth.ts リクエスト認証の唯一のエントリーポイント。
  • src/lib/sessionStore.ts LRU + Postgresによるセッション検索。DBを呼び出すのはここだけ。
  • src/server/routes/session.ts 現在のセッションをクライアントに返す。
  • src/server/routes/login.ts SessionStore.createを介して認証情報をクッキーと交換する。
  • src/app/providers/AuthProvider.tsx サーバーセッションを反映するクライアント側のコンテキスト。
  • db/migrations/004_sessions.sql sessionsテーブルとインデックスのスキーマ。

注意点

  • SessionStore内のLRUはプロセスごとに存在します。セッションを無効化してもローカルキャッシュがクリアされるだけです——他のワーカーはTTLが切れるまで最大60秒間、そのセッションを返し続ける可能性があります。
  • verifyTokenはexpiresAtをDate.now()と比較していますが、カラムの型はtimestamptzです。ドライバーがDateを返すため比較は機能しますが、チェックを調整せずに生の文字列へリファクタリングしないでください。
