04

主要コード

最も間違えやすい2つの部分は、マイグレーション（ソフトデリート、既読カーソル）と楽観的ミューテーション（一時IDの整合処理）です。

packages/db/migrations/0042_comments.sql
create table comments (
  id          uuid primary key default gen_random_uuid(),
  task_id     uuid not null references tasks(id),
  parent_id   uuid references comments(id),   -- one level only,
                                                -- enforced in API
  author_id   uuid not null references users(id),
  body        text not null,
  created_at  timestamptz not null default now(),
  deleted_at  timestamptz                       -- soft delete
);

create table comment_reads (
  task_id    uuid not null references tasks(id),
  user_id    uuid not null references users(id),
  read_up_to timestamptz not null,
  primary key (task_id, user_id)
);

create index comments_task_created
  on comments (task_id, created_at);
apps/web/hooks/useAddComment.ts
export function useAddComment(taskId: string) {
  const qc = useQueryClient();
  return trpc.comments.create.useMutation({
    onMutate: async (input) => {
      const temp = { ...input, id: `temp-${nanoid()}`,
                     createdAt: new Date(), pending: true };
      qc.setQueryData(key(taskId), (prev) =>
        [...(prev ?? []), temp]);
      return { tempId: temp.id };
    },
    onSuccess: (row, _v, ctx) => {
      // reconcile temp id → real id so the
      // realtime append doesn't duplicate it
      qc.setQueryData(key(taskId), (prev) =>
        prev.map((c) => c.id === ctx.tempId ? row : c));
    },
    onError: (_e, _v, ctx) => {
      qc.setQueryData(key(taskId), (prev) =>
        prev.filter((c) => c.id !== ctx.tempId));
    },
  });
}

リスクと対策

リスク
深刻度
対策
リアルタイム重複: ソケットの追記がHTTPレスポンスおよび一時IDの整合処理と競合する。
高
キャッシュ更新処理内で、サーバー割り当てのidを基に重複を排除する。ソケットのペイロードには本番IDが含まれ、整合処理の際に一時行はフィルタリングされる。
ユーザーが別のデバイスでスレッドを読んだ場合、未読件数が古いままになる。
中
同じチャンネル上でcomment_readsのupsertをブロードキャストする。クライアントは自身のカーソルをmax(ローカル, リモート)として扱う。
貼り付けられたMarkdown内でのメンション検出の誤検出（@media、@2xなど）。
低
書き込み時にワークスペースメンバーのみを対象にメンションを解決し、解決済みのユーザーIDを保存する——読み込み時に再解析は行わない。

未解決の課題

編集を許可するか、それとも削除して再投稿のみにするか？
編集を実装するにはedited_atカラムと「編集済み」であることを示すUIが必要になる。削除して再投稿する方法はシンプルだが、返信の紐付け（アンカー）が失われる。v1では削除のみにする方向で検討中。

メンションごとに即時通知すると、通知が多くノイズになる。提案: 15分単位でバッチ処理し、タスクごとに1件のメールへ集約し、既存の設定テーブルにある通知休止時間（quiet hours）を尊重する。
