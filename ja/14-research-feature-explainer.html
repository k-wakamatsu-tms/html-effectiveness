acme/api におけるレート制限の仕組み

TL;DR — すべてのリクエストは rateLimit() ミドルウェアを通過します。このミドルウェアは 呼び出し元をバケットキーに解決し、Redis からトークンバケットを取得して、トークンを1つ消費するか 429 を返します。上限は config/limits.yaml でルートごとに宣言され、エントリの ないルートは default ティア（APIキーごとに100リクエスト/分）を継承します。

リクエストパスをステップごとに見る

各ステップを展開すると、何が実行され、どこに実装されているかを確認できます。パス全体はおよそ40行で、 すべてのリクエストに約0.4 ms（p50）を追加します。

1 ・ 呼び出し元を識別 middleware/ratelimit.ts:21

ミドルウェアはまず、リクエストを bucketKey に変換します。Authorization ヘッダーが存在すればAPIキーを、なければ（自社のLBのみを信頼する） x-forwarded-for チェーン経由でクライアントIPを使用します。匿名のIPトラフィックには、 はるかに低いデフォルトティアが適用されます。

2 ・ バケットを参照 lib/tokenBucket.ts:9

ルート名とバケットキーを組み合わせたものが、tokens と updatedAt を保持する Redisハッシュ（rl:{route}:{key}）にマッピングされます。キーが存在しない場合は満容量で 遅延生成されます — ウォームアップはありません。

3 ・ 補充して消費 lib/tokenBucket.ts:31

補充量は経過時間から計算され（rate × Δt、上限は burst）、その後トークンが 1つ減算されます。この読み取り・更新・書き込み全体は単一のLuaスクリプトとして実行されるため、同時 リクエストによる二重消費は発生しません。

4 ・ 枯渇時に拒否 middleware/ratelimit.ts:48

スクリプトが tokens < 0 を返した場合、ミドルウェアは 429 Too Many Requests で即座に処理を打ち切り、トークンが1つ補充されるまでの秒数を Retry-After に設定します。成功したレスポンスには常に X-RateLimit-Remaining が付与されます。

ルートに上限を設定する

ミドルウェア自体を変更する必要はありません。ルート名をキーとして config/limits.yaml に エントリを追加し、（必要であれば）ミドルウェアがそのルートを見つけられるようにタグ付けするだけです。

# config/limits.yaml
default:
  rate: 100/min
  burst: 120

search.query:
  rate: 20/min
  burst: 40
  key: api_key        # or: ip
// routes/search.ts
router.post(
  "/search",
  rateLimit("search.query"),
  handler,
);
HTTP/1.1 429 Too Many Requests
Retry-After: 17
X-RateLimit-Limit: 20
X-RateLimit-Remaining: 0

{ "error": "rate_limited", "retry_after": 17 }
★
デフォルトティアだけで十分な場合は、YAMLエントリは一切不要です — ハンドラーを引数なしの rateLimit() でラップするだけです。ルート名はパスから自動的に推測されます。

知っておくべき注意点

  • 開発環境では上限がプロセス単位になります。 REDIS_URL が 未設定の場合、Redisクライアントはインメモリのマップにフォールバックするため、ローカルテストは実際の クラスターの挙動を反映しません。
  • Burst と rate は同じではありません。 burst はバケットの 容量であり、1分間アイドル状態だった呼び出し元は、rate が低くても burst 件の リクエストを瞬時に送信できます。
  • ストリーミングレスポンスは1回としてカウントされます。 トークンはリクエスト開始時に消費される ため、30秒間のSSEストリームであってもコストはトークン1つ分です。

よくある質問

内部トラフィックを除外するにはどうすればよいですか？
呼び出し元から x-acme-internal: 1 を設定してください。ミドルウェアはこれをmTLSの ピア名と照合し、バケット処理を完全にスキップします。
誰が制限を受けているかはどこで確認できますか？
429 が発生するたびに、ルートとキー種別のタグが付いた ratelimit.rejected メトリクスが送出されます。API → Health 配下にGrafanaパネルがあります。
特定のユーザーだけ上限を高くできますか？
できます — YAML内の overrides: に該当ユーザーのAPIキーを追加してください。 オーバーライドはデプロイなしで再読み込みされます。
