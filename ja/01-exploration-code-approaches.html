デバウンス検索を実装する3つの方法

プロンプト 私たちの React コードベースにあるタスクフィルター入力欄に、デバウンス検索を実装する 3つの異なる方法を、それぞれのトレードオフとともに教えてください。

01インラインの useEffect + setTimeout

デバウンス処理を、入力欄を持つコンポーネント自体の中に直接記述する。

export function TaskSearch() {
  const [draft, setDraft] = useState('');
  const [query, setQuery] = useState('');

  useEffect(() => {
    const id = setTimeout(() => setQuery(draft), 300);
    return () => clearTimeout(id);
  }, [draft]);

  const { data } = useTasks({ search: query });

  return (
    <input
      value={draft}
      onChange={(e) => setDraft(e.target.value)}
      placeholder="Filter tasks…"
    />
  );
}

デメリット
新たに学ぶべき抽象化が一切ない
検索が存在する箇所すべてでロジックが重複する
devtools でステップ実行しやすい
1つの概念的な値に対して2つの状態を持つことになる

遅延の定数値がコンポーネント本体に埋もれてしまう

02カスタム useDebounce フック

タイマー処理を src/hooks/ 配下の共有フックに切り出す。

// src/hooks/useDebounce.ts
export function useDebounce<T>(value: T, ms = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(id);
  }, [value, ms]);
  return debounced;
}

// TaskSearch.tsx
const [draft, setDraft] = useState('');
const query = useDebounce(draft, 300);
const { data } = useTasks({ search: query });
メリット
デメリット
フィルター、コマンドバー、ボード検索など全体で1つの import を再利用できる
保守・ドキュメント化すべきファイルが1つ増える
fake timers を使えば単体テストが容易
ジェネリックの T が意図をやや隠してしまう
遅延時間が目に見える、調整可能な引数になっている
それでもキー入力のたびに再レンダリングされる
バンドルへの影響: +0.2 kb テスト容易性: 高 再利用性: 高 SSR 安全性: あり

03小さな外部ライブラリ

値とコールバックの両方に対応する use-debounce を採用する。

import { useDebouncedCallback }
  from 'use-debounce';

export function TaskSearch() {
  const [query, setQuery] = useState('');

  const onChange = useDebouncedCallback(
    (next: string) => setQuery(next),
    300,
    { leading: false, maxWait: 1000 },
  );

  const { data } = useTasks({ search: query });

  return (
    <input
      defaultValue=""
      onChange={(e) => onChange(e.target.value)}
    />
  );
}
メリット
デメリット
leading / trailing / maxWait を代わりに処理してくれる
監査・更新が必要な新しいランタイム依存が増える
コールバック形式なので中間の再レンダリングを省ける
非制御コンポーネントは Acme のフォームパターンから外れる
アンマウント・flush・キャンセルなどのエッジケースが十分にテストされている
自前で持てる程度のものに gzip 圧縮後で~1.4 kb かかる
バンドルへの影響: +1.4 kb テスト容易性: 高 再利用性: 高 SSR 安全性: あり

推奨

アプローチ02、カスタム useDebounce フックを採用しましょう。 Acme にはすでに、インラインパターンを手書きしている箇所が3つあり (タスクフィルター、コマンドパレット、メンバーピッカー)、 共有フックを1つ切り出すことで、新しい依存関係を持ち込まずに重複を解消できます。

アプローチ03を再検討するのは、後になって maxWait や flush() のセマンティクスが必要になった場合だけです — 要件が10行程度のフックの枠を超えて拡大したときに初めて、 このライブラリはそのバンドルコストに見合う価値を発揮します。
